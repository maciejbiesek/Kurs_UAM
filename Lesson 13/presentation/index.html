Unit tests?

  • Automated set of conditions that check your code
  • Defines your specification and makes sure that your code meets it
  • You can quickly check if your code works properly
  • Used mostly for low level functionalities

TDD

Focus on tests, write them at first, then write the code

  • Write first test (test fails - is red)
  • Write the code that passes this one test (green)
  • Write second test (red)
  • Write the code for the second test (green)
  • Refactor when needed
  • ???
  • Profit

Why? You focus on the goal, on the specification, not on the implementation

Theoretically, it gives you the simplest, working solution

Tests structure

  • describe - Wrap the set of related tests
  • it - single test, should be as simple as possible
  • expect(va).toBe(3) - single assertion
  • describe - Adding two values
    • it - Should add two numbers
    • it - Should add two negative numbers
    • it - Should throw if provided with wrong argument type
  • describe - Dividing values
    • it - Should divide the numbers
    • it - Should throw when dividing by zero
etc.

Assertions

Simple comparisons


expect(value).toBe("something"); //strict comparison "==="
expect(value).toEqual([1, 2, 3]); //deep comparison, compare the whole structure
expect(value).not.toBe('smth');
expect(value).not.toEqual();

expect(12).toEqual(jasmine.any(Number));

Exceptions


expect(func).toThrow();
expect(function () {
    calc.divide(5, 0);
}).toThrow();
expect(func).not.toThrow();

Spies

We can spy on methods, functions, to check the behavior


var helpers;
beforeEach(function () {
    helpers = {
        countSmth: jasmine.createSpy('countSmth').and.returnValue(666);
    };
});

it('Should call the helper', function () {
    expect(helpers.countSmth).toHaveBeenCalled();
    expect(helpers.countSmth.calls.count()).toBe(1);
});
